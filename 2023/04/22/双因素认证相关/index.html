双因素认证

1. 双因素认证介绍

1.1. 引言

所谓的认证（authentication）就是确认用户的身份，而在认证中，最常见的方式就是密码，但这并不安全，拥有密码的人并不一定是用户本身。所以，越来越多的地方要求使用双因素认证（two-factor authentication, 2FA）。

1.2. 2FA 概念

一般来说，3 种不同类型的证据，可以证明一个人的身份：

  1. 秘密信息：只有用户知道的，其他人不知道的信息，比如密码；
  2. 个人物品：用户的私人物品，比如身份证、钥匙；
  3. 生理特征：用户的遗传特征，比如指纹、虹膜等；

以上 3 种证据称为 3 种因素，因素越多，证明力越强，身份越可靠。而双因素认证就是在认证时同时认证上面的 2 种因素，最常见的就是去银行办业务，用户必须提供银行卡（个人物品）和密码（秘密信息）。

常用的双因素组合是“密码+某种个人物品”，比如网上银行的 U 盾（时代的眼泪），用户在电脑上插入 U 盾，然后输入密码才可以登录网上银行。问题是，用户不会随身携带 U 盾，但会随身携带手机，所以 “密码+手机”是现在最佳的双因素认证方案。

现在很多网站第一次登录的时候，除了需要用户名和密码，大多都会给绑定的手机发送验证码进行验证，这就是双因素认证，但其实这也是不安全，所以就有了更安全的双因素认证：TOTP。

1.3. Time-based One-time Password

TOTP 名字直译过来基于时间的一次性密码。这是公认的可靠解决方案，已写入国际标准 RFC 6238 - TOTP: Time-Based One-Time Password Algorithm

TOTP 的步骤如下：

  1. 用户开启双因素认证，服务器生成一个密钥；

  2. 服务器需要把密钥给用户，常见的比如扫描二维码，用户扫描过后，就把密钥保存到手机中，此时服务器和手机拥有了同一把密钥；

  3. 用户登录时，手机客户端会使用这个密钥和当前 Unix 时间戳生成一个 hash，该 hash 默认时间是 30 秒，用户必须在有效期内把 hash 提交给服务器；

  4. 用户提交后，服务器也会使用密钥和当前时间戳生成一个 hash，与用户提交的 hash 作对比，如果相同则登录成功，否则拒绝登录；

有一点需要注意：密钥跟手机绑定，一旦更换收集也需要更换密钥。

Google 身份验证器就是使用 TOTP。

上述步骤有一个问题，用户提交 hash 之后服务器才生成 hash，那么手机和服务器的时间戳是不同的，为什么 hash 会相同呢？这是因为 hash 生成的公式。下面是 hash 生成公式的重要部分：

\[ TC = floor(unixtime(now)-unixtime(T0))/TS \]

  • TC：时间计数器 time counter；
  • \(unixtime(now)\)：当前的 Unix 时间戳；
  • \(unixtime(T0)\)：起始时间的 Unix 时间戳，默认是 0，也就是 1970 年 7 月 1 日；
  • TS：哈希有效期，默认是 30 秒；

因此，该公式变为：

\[ TC = floor(unixtime(now))/30 \]

因此，只要是 30 秒了，两者的 TC 值相同，然后利用 TC 去计算 hash：

\[ TOTP\_hash = HASH(secretKey, TC) \]

TC 相同，所以计算出的 hash 也相同。

1.4. 总结

双因素认证的优点在于，比单纯的密码登录安全得多。就算密码泄露，只要手机还在，账户就是安全的。各种密码破解方法，都对双因素认证无效。

缺点在于，登录多了一步，费时且麻烦，用户会感到不耐烦。而且，它也不意味着账户的绝对安全，入侵者依然可以通过盗取 cookie 或 token，劫持整个对话（session）。

双因素认证还有一个最大的问题，那就是帐户的恢复。一旦忘记密码或者遗失手机，想要恢复登录，势必就要绕过双因素认证，这就形成了一个安全漏洞。除非准备两套双因素认证，一套用来登录，另一套用来恢复账户。

2. 2FA 绕过

2.1. 使用传统 session 管理绕过

此方法是关于使用密码重置功能绕过双因素身份验证机制。在几乎所有的 web 应用程序中，密码重置功能会在重置完成后自动将用户登录到应用程序中。这就意味着重置密码的 token 和重置完成后登录的 token 是一样的，那么就可以这么利用：

  1. 申请重置密码；
  2. 获得重置密码的 token；
  3. 使用重置密码的 token 登录；

2.2. 使用 OAuth 绕过 (多半不适用)

正如所观察到的，在 OAuth 流程中没有 2fa 的干预。攻击者可能会利用此机制并利用 OAuth 登录 Web 应用程序，而不是使用用户名和密码来登录

注意：要使此绕过起作用，攻击者必须有权访问 OAuth 帐户才能代表用户登录。

过程：

  1. site.com 向 Facebook 申请 OAuth token；
  2. Facebook 验证账户（这里攻击者需要知道目标 facebook 的账号和密码）；
  3. Facebook 发送 callback 代码；
  4. 登录到 site.com

2.3. 暴力破解

通常，双因素认证代码的长度是4到6个字符，且为数字，通过暴力破解是可行的，前提是目标应用不存在请求速率限制。

2.4. 通过竞争条件绕过

攻击者可以利用以前使用过或未使用过的 token 来验证设备。

然而，这种技术要求攻击者能够访问之前生成的值，这可以通过逆向方法生成应用程序的算法或拦截之前已知的代码来实现。

(没理解为什么叫竞争条件，我认为更像是 token 可以重复使用导致的)

2.5. 替换响应绕过

步骤：

  1. 输入正确的 OTP（one-time password）；
  2. 拦截响应并丢弃；
  3. 输入错误的 OTP；
  4. 拦截响应并将其改为第 2 步中成的响应；
  5. 登录成功；

2.6. 使用激活链接（activation link）绕过

有些 Web 应用能够使用激活链接登录（有些激活链接存在漏洞，token 未及时过期）。

2.7. 密码重置页面绕过

首先进入密码重置界面，该页面可能不存在请求限制，因此可以进行暴力破解。

2.8. 使用备份验证码的相关请求绕过

假设：在某次测试中，在某个帐户设置页面看到了用于获取备份验证码的 API。

在这种场景下（成功登录），可以成功发起上述 API 请求。那么当处于另一个场景（只提供有效的电子邮件和密码但无双因素认证代码）时触发相同的 API 请求，会发生什么事？

迅速注销登录，并使用有效的电子邮件和密码再次登入。不出所料，bountyplease.com 将重定向到双因素认证页面，这次输入错误的 OTP 并捕获了请求，并做了以下两个更改：

  1. 替换原始目的地址为 POST /API/totp_auth HTTP/1.1
  2. 替换原始参数为 {“action”:”backup_codes”,”clusterNum”:”000",”accountId”:”test123",”email”:”test123@gmail.com”}

很快在响应中得到了所有的备份验证码。攻击者可以将这些代码输入双因素认证中，进入受害者的帐户。

2.9. 通过禁用 2FA 的 CSRF 攻击绕过

某次测试中试图使用 CSRF 攻击来禁用双因素认证，发现这个 CSRF POC 中存在 token。当尝试使用这个 POC 攻击时，发现这个 token 永远不会过期，可以一次又一次禁用另一个帐户上的双因素认证。

注意：或许对每个目标都可尝试使用 CSRF 来禁用双因素认证。

步骤：

  1. 注册两个帐户;
  2. 登录到攻击者帐户并捕获与禁用双因素认证有关的请求;
  3. 据此生成用于 CSRF 攻击的 HTML 页面；
  4. 登录到受害者帐户并访问恶意页面；
  5. 在受害者不知情的情况下禁用了双因素认证，从而成功绕过；
