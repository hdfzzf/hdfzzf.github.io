OAuth2.0协议

hdfzzf
2023-04-16
OAuth › 网络协议
OAuth

官网：OAuth Community Site

1. 概述与应用场景

概述：

OAuth 是一个关于授权（authorizaation）的开放网络标准，在全世界得到广泛的应用，目前的版本是 2.0。

OAUTH 协议为用户资源的授权提供了一个安全的、开放而又简易的标准。与以往的授权方式不同之处是 OAUTH 的授权不会使第三方触及到用户的帐号信息（如用户名与密码），即第三方无需使用用户的用户名与密码就可以申请获得该用户资源的授权，因此 OAUTH 是安全的。同时，任何第三方都可以使用 OAUTH 认证服务，任何服务提供商都可以实现自身的 OAUTH 认证服务，因而 OAUTH 是开放的。业界提供了 OAUTH 的多种实现如 PHP、JavaScript、Java、Ruby 等各种语言开发包，大大节约了程序员的时间，因而 OAUTH 是简易的。目前互联网很多服务如 Open API，很多大头公司如 Google，Yahoo，Microsoft 等都提供了 OAUTH 认证服务，这些都足以说明 OAUTH 标准逐渐成为开放资源授权的标准。

应用场景：

有这么一个“云冲印”网站，可以将用户储存在 Google 的照片冲印出来，用户为了使用该服务，就必须让“云冲印”读取储存在 Google 的照片。

问题出现了，“云冲印”必须得到用户的授权，才可以去读取 google 上储存的照片，那“云冲印”要如何才能获取用户的授权呢？传统的方法是用户将自己的 Google 账号、密码告诉“云冲印”，后者使用这些信息去 Google 读取照片。这样虽然能够解决问题，但会引入更多的问题：

  1. "云冲印"为了后续的服务，会保存用户的密码，即使明确说明不会保存，但没人能够保证，因此这样很不安全；
  2. Google 必须部署密码登录（Google 使用双因素认证），而单纯使用密码登录是不安全的；
  3. “云冲印”拥有账号和密码，就相当于获得了用户在 Google 的所有权限，且用户没法限制；
  4. 如果用户需要收回权限，就必须修改密码，这样一来，其他获得授权的第三方服务全部失效；
  5. 如果一个第三方服务出现漏洞，泄露了密码，那么用户的 Google 密码也被泄露；

因此，需要一种方法来提供这样场景下的解决方案，这就是 OAuth。

2. 历史：OAuth 1.0

OAuth 1.0 中定义了 3 种角色：Service Provider、User、Consumer。其中，Google 就是 Service Provider，“云冲印”就是 Consumer。

但 User 使用 Consumer 的服务时，OAuth 1.0 的工作流程开始：

  1. Consumer 向 service provider 申请 request token；
  2. Consumer 将 user 重定向到 service provider 处，user 带着未授权的 request token；
  3. User 在 service provider 同意授权；
  4. Service provider 将 user 重定向回 consumer，user 带回授权过的 request token；
  5. Consumer 使用授权过的 request token 向 service provider 交换 access token；
  6. Consumer 使用 access token 访问 user 在 service provider 处储存的资源；

概念介绍：

  • Request token：consumer 用于取得 user 授权的载体，有 service provider 提供，请求用户授权；
  • Access token：consumer 访问 user 储存在 service provider 处的资源，可以使用授权过的 request token 与 service provider 交换 access token；

工作流程中主要的请求、响应参数：

第 1步：请求 request token；

请求参数 响应参数
oauth_consumer_key：consumer 标识符 oauth_token：未授权的 request token
oauth_signature_method：签名方法 oauth_token_secret：与 requesttoken 匹配的密码
oauth_signature：签名值
oauth_timestamp：时间戳
oauth_nonce：随机值，唯一值，防止重放攻击
oaut_version：版本，填 1.0

第 2 步：重定向 user 到 service provider，此时只有请求参数；

  1. Oauth_token；
  2. Oauth_callback；
  3. 其他；

第 3 步：用户授权，这一步不好说，每个 service provider 要求的可能都不同，但请求参数大都有：

  1. Service provider 要求 user 提供证明信息，比如用户名和密码；
  2. Service provier 向 user 说明：要授权哪些信息、要授权给哪个 consumer 等；
  3. User 在知晓之后主动进行授权，授权的访问也由用户指定，service provider 保护没有授权的部分不允许 consumer 访问；

第 4 步：user 带着授权的 request token 返回 consumer；

  • User 授权完毕之后，service provider 根据第 2 步中的 oauth_callback 的值将 user 重定向到 consumer，user 回 consumer 的请求包中携带有 oauth_token，用来表名是否授权成功；

第 5 步：获取 access token；

请求参数 响应参数
oauth_consumer_key oauth_token：此时是 access token
oauth_token：经授权的 requet token oauth_token_secret：与 access token 匹配的密码
oauth_signature_method
oauth_signature
oauth_timestamp
oauth_nonce
oauth_version

第 6 步：访问 service provider 上的资源，请求参数如下；

  1. Oauth_consumer_key
  2. Oauth_token，也就是第 5 步中取回的 access token；
  3. Oauth_signature_method
  4. Oauth_sighature
  5. Oauth_timestamp
  6. Oauth_nonce
  7. Oauth_version

Oauth 1.0 存在一个严重的安全问题，攻击流程如下：

  1. Attacker 登录 consumer，发起 oauth 授权，这时候 consumer 去 service provicer 申请 request token；
  2. Consumer 重定向 attakcer 到 service provider，但这个请求包被拦截，attacker 将这个 url 通过钓鱼的方式让受害者点击；
  3. 受害者完成授权，service provider 将受害者重定向到某个不知名的地方（由攻击者的 url 中的 callback 指定）；
  4. 此时，攻击者也同时获得了授权的 request token（受害者授权的），使用 request token 与 service provider 交换 access token，从而可以获取受害者的资料；

上述的攻击过程，受害者的密码没有泄露，任何信息都没有泄露、修改，唯一的问题是攻击者钓鱼使受害者主动完成了授权。

这事因为 oauth 1.0 过程中，没有验证是否是同一个用户触发的整个流程，service provider 和 consumer 只根据 request token 来判断用户是否授权，不在乎是哪个用户。

经过这次安全漏洞事件（2009 年上面的漏洞被披露），OAuth 协议发布了修正的版本，包括很多小的错误更正，就是后来的广为使用的 OAuth 1.0 Core Revision A，一般称为 OAuth 1.0a，也是 Twitter 至今仍在使用的版本。

在 oauth 1.0 的基础上做了一些修改：

  1. 请求 request token 的时候需要传递 oauth_callback，该参数的作用是 consumer 告诉 service provider，如果 user 授权完成后，就把 user 重定向到 oauth_callback 处。好处：攻击者即使拿到授权的 request token，因为无法知道 oauth_callback，那就没办法使用 request token。Service provier 对于该请求的响应中包含了 oauth_callback_confimed。
  2. 重定向 user 到 service provider 中取消了 oauth_callback；
  3. 用户授权完毕后，service provier 将 user 重定向到第一步中的 oauth_callback 指定的 url，user 除了带回授权后的 request token，还增加了 oauth_verifier；
  4. Consumer 使用 requet token 交换 access token 时，需要验证 oauth_verifier，该参数能验证授权的用户和登录 consumer 的用户是否是同一个；

3. Oauth 2.0

3.1. 名词定义

  1. Third-party application：第三方应用程序，也叫作客户端（client），比如 1. 概述与应用场景 中的云冲印；
  2. HTTP Service：HTTP 服务提供商，比如 1. 概述与应用场景 中的 google；
  3. Resource Own：资源所有者，也就是用户（user）；
  4. User Agent：用户代理，一般就是浏览器；
  5. Authorization Server：认证服务器，服务提供商专门用来处理认证的服务器；
  6. Resource Server：资源服务器，服务提供商用来存放用户数据的服务器；

Oauth 的作用就是：让客户端安全可控地获取用户的授权，与服务提供商进行互动。

3.2. Oauth 的思路与运行流程

OAuth 在"客户端"与"服务提供商"之间，设置了一个授权层（authorization layer）。"客户端"不能直接登录"服务提供商"，只能登录授权层，以此将用户与客户端区分开来。"客户端"登录授权层所用的令牌（token），与用户的密码不同。用户可以在登录的时候，指定授权层令牌的权限范围和有效期。

"客户端"登录授权层以后，"服务提供商"根据令牌的权限范围和有效期，向"客户端"开放用户储存的资料。

Oauth 2.0 的运行流程如下：

用户打开客户端以后，上述流程开始。步骤解释：

  • A：客户端要求用户给予授权；
  • B：用户同意给客户端授权；
  • C：客户端使用上一步获得的授权，向认证服务器申请 token；
  • D：认证服务器对客户端进行认证，通过以后，同意发放 token；
  • E：客户端使用 token，向资源服务器申请获取资源；
  • F：资源服务器确认令牌无误，同意向客户的开放资源；

上面的步骤中，最难也是最关键的就是 B，用户该如何给客户端授权呢？

3.3. 客户端的授权模式

客户端必须得到用户的授权（authorization grant），才能获得令牌（access token），oauth 2.0 定义了 4 中授权方式：

  1. 授权码模式（authorization code）；
  2. 简化模式（implicit）；
  3. 密码模式（resource owner password credentials）；
  4. 客户端模式（client credentials）；

3.3.1. 授权码模式

授权码模式（authorization code）是功能最完整、流程最严密的授权模式。它的特点就是通过客户端的后台服务器，与"服务提供商"的认证服务器进行互动。

用户登录客户端之后，流程开始。步骤（步骤中用户和代理都叫用户）：

  • A：客户端要求用户给予授权，将用户重定向到认证服务器；
  • B：用户选择是否给予客户端授权；
  • C：假设上一步，用户选择了授权，那么认证服务器就会将用户重定向到客户端事先指定的 uri，同时用户会携带授权码；
  • D：客户端收到授权码，附上早先指定的重定向 uri，向认证服务器申请 access token，这一步在客户端的后台服务器上完成，对用户透明；
  • E：认证服务器核对了授权码和重定向 uri，确认无误后向客户端发送 access token 和 refresh token；

A 步骤中，客户端申请认证的 URI，包含以下参数：

  • response_type：表示授权类型，必选项，此处的值固定为"code"
  • client_id：表示客户端的ID，必选项
  • redirect_uri：表示重定向URI，可选项
  • scope：表示申请的权限范围，可选项
  • state：表示客户端的当前状态，可以指定任意值，认证服务器会原封不动地返回这个值。

C 步骤中，服务器回应客户端的 URI，包含以下参数：

  • code：表示授权码，必选项。该码的有效期应该很短，通常设为10分钟，客户端只能使用该码一次，否则会被授权服务器拒绝。该码与客户端ID和重定向URI，是一一对应关系。
  • state：如果客户端的请求中包含这个参数，认证服务器的回应也必须一模一样包含这个参数。

D 步骤中，客户端向认证服务器申请令牌的 HTTP 请求，包含以下参数：

  • grant_type：表示使用的授权模式，必选项，此处的值固定为"authorization_code"。
  • code：表示上一步获得的授权码，必选项。
  • redirect_uri：表示重定向URI，必选项，且必须与A步骤中的该参数值保持一致。
  • client_id：表示客户端ID，必选项。

E 步骤中，认证服务器发送的 HTTP 回复，包含以下参数：

  • access_token：表示访问令牌，必选项。
  • token_type：表示令牌类型，该值大小写不敏感，必选项，可以是bearer类型或mac类型。
  • expires_in：表示过期时间，单位为秒。如果省略该参数，必须其他方式设置过期时间。
  • refresh_token：表示更新令牌，用来获取下一次的访问令牌，可选项。
  • scope：表示权限范围，如果与客户端申请的范围一致，此项可省略。

相关参数使用 JSON 发送，并且 HTTP 头信息明确指定不允许缓存。

3.3.2. 简化模式

简化模式（implicit grant type）不通过第三方应用程序的服务器，直接在浏览器中向认证服务器申请令牌，跳过了"授权码"这个步骤，因此得名。所有步骤在浏览器中完成，令牌对访问者是可见的，且客户端不需要认证。

用户登录客户端后，上述流程开启：

  • A：客户端将用户重定向到认证服务器；
  • B：用户决定是否给予授权；
  • C：假设上一步中同意授权，认证服务器将将用户重定向到客户端指定的 uri ，并在 uri 的 hash 部分包含了 access token；
  • D：浏览器向资源服务器发出请求，不包含上一步中得到的 access token；
  • E：资源服务器返回一个网页，其中包含的代码可以获取 hash 中的 access token；
  • F：浏览器执行上一步获得的 script，得到 access token；
  • G：浏览器将 access token 发给客户端；

A 步骤中，客户端发出的 HTTP 请求，包含以下参数：

  • response_type：表示授权类型，此处的值固定为"token"，必选项。
  • client_id：表示客户端的ID，必选项。
  • redirect_uri：表示重定向的URI，可选项。
  • scope：表示权限范围，可选项。
  • state：表示客户端的当前状态，可以指定任意值，认证服务器会原封不动地返回这个值。

C 步骤中，认证服务器回应客户端的 URI，包含以下参数：

  • access_token：表示访问令牌，必选项。
  • token_type：表示令牌类型，该值大小写不敏感，必选项。
  • expires_in：表示过期时间，单位为秒。如果省略该参数，必须其他方式设置过期时间。
  • scope：表示权限范围，如果与客户端申请的范围一致，此项可省略。
  • state：如果客户端的请求中包含这个参数，认证服务器的回应也必须一模一样包含这个参数。

在上面的例子中，认证服务器用 HTTP 头信息的 Location 栏，指定浏览器重定向的网址。注意，在这个网址的 Hash 部分包含了令牌。

根据上面的 D 步骤，下一步浏览器会访问 Location 指定的网址，但是 Hash 部分不会发送。接下来的 E 步骤，服务提供商的资源服务器发送过来的代码，会提取出 Hash 中的令牌。

3.3.3. 密码模式

密码模式（Resource Owner Password Credentials Grant）中，用户向客户端提供自己的用户名和密码。客户端使用这些信息，向"服务商提供商"索要授权。

在这种模式中，用户必须把自己的密码给客户端，但是客户端不得储存密码。这通常用在用户对客户端高度信任的情况下，比如客户端是操作系统的一部分，或者由一个大公司出品。而认证服务器只有在其他授权模式无法执行的情况下，才能考虑使用这种模式。

步骤：

  • A：用户向客户端提供用户名和密码；
  • B：客户端使用用户名和密码向认证服务器申请 access token；
  • C：认证服务器确认无误后，向客户端提供 access token；

B 步骤中，客户端发出的 HTTP 请求，包含以下参数：

  • grant_type：表示授权类型，此处的值固定为"password"，必选项。
  • username：表示用户名，必选项。
  • password：表示用户的密码，必选项。
  • scope：表示权限范围，可选项。

C 步骤中，认证服务器向客户端发送访问令牌。

整个过程中，客户端不得保存用户的密码。

3.3.4. 客户端模式

客户端模式（Client Credentials Grant）指客户端以自己的名义，而不是以用户的名义，向"服务提供商"进行认证。严格地说，客户端模式并不属于 OAuth 框架所要解决的问题。在这种模式中，用户直接向客户端注册，客户端以自己的名义要求"服务提供商"提供服务，其实不存在授权问题。

步骤：

  • A：客户端向认证服务器进行身份认证、申请 access token；
  • B：认证服务器确认无误后，向客户端提供 access token；

A 步骤中，客户端发出的 HTTP 请求，包含以下参数：

  • grant_type：表示授权类型，此处的值固定为"client_credentials"，必选项。
  • scope：表示权限范围，可选项。

B 步骤中，认证服务器向客户端发送访问令牌。

3.4. 更新 token

如果用户访问的时候，客户端的"访问令牌"已经过期，则需要使用"更新令牌"申请一个新的访问令牌。

客户端发出更新令牌的HTTP请求，包含以下参数：

  • grant_type：表示使用的授权模式，此处的值固定为"refresh_token"，必选项。
  • refresh_token：表示早前收到的更新令牌，必选项。
  • scope：表示申请的授权范围，不可以超出上一次申请的范围，如果省略该参数，则表示与上一次一致。
